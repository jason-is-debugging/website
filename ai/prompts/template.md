# code requirements

单一职责原则 (SRP)：每个函数/方法行数严格控制在 30 行以内，每个文件行数控制在 200 行以内。拒绝面条代码。

拒绝冗长：优先使用语言的高级特性（如 Go 的 defer、Vue3 的 Composable）和内置函数来精简逻辑。

设计模式至上：在合适场景下必须使用设计模式（如后端工厂模式、策略模式、单例模式；前端发布订阅、装饰器等），并在代码注释中明确标出。

DRY 原则：任何重复出现 2 次以上的逻辑，必须抽象为公共函数、中间件或组件。

# all

角色：你是一位精通 Go (后端) 和 Vite + Vue3 + TS (前端) 的全栈架构专家，追求极致的“小而精”代码美学。
任务：请为我同时实现 [具体功能模块名称，例如：商品评论系统] 的全栈前后端代码。

为了保证代码短小精悍并应用设计模式，请严格按照以下【四个步骤】按顺序输出：

---

### 第一步：契约定义（API 协议）
请先给出前后端交互的 RESTful API 接口定义，使用 JSON 格式。包含：
- 请求路由、方法
- 请求体 (Request Body) 和 响应体 (Response Body) 的最小化字段定义

### 第二步：后端实现 (Go - 小而精)
请编写后端的业务逻辑，要求：
1. 架构分层：只需给出 Controller 层和 Service 层的核心代码。
2. 设计模式：使用合理的来处理核心业务逻辑。
3. 严苛限制：每个函数严禁超过 25 行，利用 Go 的错误提前返回减少 else 嵌套，拒绝任何臃肿逻辑。

### 第三步：前端实现 (Vue3 + TS - 逻辑分离)
请编写前端对应的 UI 与逻辑，要求：
1. 类型定义：用 TS 严格定义与后端对接的 API 响应接口。
2. 状态与逻辑抽离：必须将 API 请求、状态管理提取到一个独立的 Composable 函数中（useXXXX.ts）。
3. 视图层组件：Vue 组件（.vue）只负责渲染和调用 Composable，行数控制在 100 行以内。

### 第四步：解耦与重构说明
请用两句话分别说明：
1. 后端代码中哪里的设计模式体现了高扩展性？
2. 前后端是如何通过“单一职责”做到文件小而精的？
